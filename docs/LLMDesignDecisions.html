1. Why did you choose this LLM?
LLM selection journey — from Groq to Google Gemini 2.5 Flash
💡
Short answer: We started with Groq (compound-beta / LLaMA), hit severe free-tier throttling, and switched to Google Gemini 2.5 Flash for its 1M token context window, generous rate limits, and consistent structured output quality.

LLM Comparison — Why Gemini Won

Criterion Groq (compound-beta / LLaMA) Google Gemini 2.5 Flash
Context window 131K tokens 1,048,576 tokens (1M)
Free tier RPM 30 RPM (compound-beta) 15 RPM free / 1,000 RPM Tier 1
Free tier TPM 70,000 TPM (compound-beta) 1,000,000 TPM free / 4,000,000 Tier 1
Max output tokens 8,192 tokens 65,536 tokens
Structured JSON output Inconsistent — required 6 fallback strategies Reliable with well-defined system prompt
Streaming support compound-beta returned empty on streaming Non-streaming REST API — stable and predictable
Cost per DocuGenious request Free (but heavily throttled) ~$0.01 per generation (Tier 1)
Cloud deployment support Issues with Azure App Service env Pure REST — works everywhere
Rate-limit retry info Limited Returns retryDelay in JSON error body
🚫 Why Groq Didn't Scale
  • compound-beta returned empty responses on streaming — had to switch to non-streaming
  • LLaMA free tier: only 6,000 TPM — barely enough for 1 ticket
  • Dynamic token cap needed to avoid HTTP 413 errors on free tier
  • U+2028/U+2029 Unicode line separators caused silent JSON parse failures
  • Required 6 fallback JSON parsing strategies to handle inconsistent output
✅ Why Gemini 2.5 Flash
  • 1M token context window — entire codebases fit in one call
  • Consistent structured JSON when given a clear system prompt
  • retryDelay field in 429 error body enables precise retry timing
  • Simple REST API — no SDK dependency, works on any runtime
  • 4M TPM on Tier 1 — effectively unlimited for a documentation tool
  • Single model covers all document types without switching

2. Key Learnings
Real challenges encountered and how they were resolved across 20+ commits
  • 1
    Streaming API ≠ Reliable Output
    compound-beta's streaming endpoint returned empty responses silently. Switching to a single non-streaming POST call resolved it immediately. Lesson: always validate streaming vs non-streaming behaviour independently for each model.
    Switch from streaming to non-streaming API call
  • 2
    LLMs Produce Inconsistent JSON — Build Fallback Pipelines
    Even with explicit JSON instructions, Groq returned malformed collections, nested objects instead of arrays, and raw strings where lists were expected. Six fallback strategies were implemented: direct parse → lenient parse → field extraction → flatten → strip markdown → last-resort recovery. Lesson: never trust a single JSON parse path with LLM output.
    Add flexible JSON converters for collection fields
  • 3
    Unicode Line Separators Silently Break JSON
    U+2028 (Line Separator) and U+2029 (Paragraph Separator) are valid in JavaScript strings but illegal in JSON strings per RFC 8259. LLMs sometimes emit them in generated text, causing JsonException at parse time with no obvious error message. Lesson: sanitise LLM output for invisible Unicode characters before parsing.
    Fix JSON parse failure caused by U+2028/U+2029 Unicode line separators
  • 4
    Truncation Is Critical — And Must Be Configurable
    Without truncation, a single large Git repo could exceed context limits or generate enormous token costs. Initially hardcoded at 2,000 chars/file — later made fully configurable via appsettings.json so operators can tune the quality vs cost trade-off without code changes. Current production values: 5,000 chars/file, 100,000 char prompt cap.
    Make Gemini context truncation limits configurable via appsettings
  • 5
    Git Clone Fails on Cloud Environments Without Temp Directory
    On Azure App Service, the application directory is read-only. Cloning into a relative path (e.g. ./clones) caused permission failures at runtime. Fix: use Path.GetTempPath() with a unique subdirectory per clone. This also works on Docker, Linux, and any CI environment. Lesson: always use the OS temp directory for ephemeral file operations in cloud deployments.
    Fix Git clone failure on Azure App Service — use temp directory
  • 6
    Rate Limit Retries Need Precise Timing
    Gemini embeds the exact wait duration in the 429 error body as error.details[].retryDelay (e.g. "4.197s"). Parsing this and waiting exactly that long + 1s buffer gave much higher retry success rates than a fixed 5s sleep. Multiple formats handled: "4s", "4.197184507s", Retry-After header, and inline error message text.
    ExtractRetryDelay / ParseRetryAfterSeconds implementation
  • 7
    UI State Machine Eliminates Race Conditions
    Early versions had scattered boolean flags (_isLoading, _hasError, _showResult) that could reach impossible states. Replacing them with a single UiState enum (Form → Validating → Generating → Success/Error) made transitions explicit and eliminated entire categories of bugs including the "Close resets document type" issue.
    Fix Document Type resetting to wrong value on Close/Back to form
  • 8
    Branding in Status Messages Leaks Implementation Details
    Live status messages said "Analysing with Gemini AI…". Users interact with Docu-Genius, not Gemini — exposing the underlying model is a branding concern and creates upgrade friction if the LLM changes. Changed to "Analysing and generating…". Lesson: abstract LLM identity from user-facing messages from day one.
    Polish UI: remove AI branding from status message

3. How Did You Validate the Output?
Multi-layer validation from input to final PDF
Layer 1 — Input Validation
Before any API call is made, the Blazor UI + ValidationService verify source type, ticket IDs, repo URL, and document type. User Guide requires all tickets to be in Done/Complete status.
Layer 2 — AI Output Normalisation
After Gemini responds, NormalizeResult() strips placeholder text and empty arrays. RedistributeSections() rescues content that was placed in the wrong field.
Layer 3 — PDF Rendering
QuestPDF renders each section conditionally — empty fields are silently skipped, so the PDF is never padded with blank sections regardless of what the AI returns.
🔍
Input Validation (ValidationService) — Checks run live in the modal before generation starts:
  • Jira connection reachable + credentials valid
  • Each ticket ID exists and is accessible
  • Git repository URL is reachable and the branch exists
  • Output file name is safe (no path traversal characters)
  • Document type is valid — User Guide blocked if any ticket is not Done
🤖
AI Output Normalisation (GeminiService) — Applied to every AnalysisResult:
  • NormalizeResult() — removes filler phrases like "N/A", "Not applicable", empty brackets
  • RedistributeSections() — scans executive summary for ## Heading markers and moves them to the correct field
  • 6-strategy JSON fallback pipeline — handles malformed output, nested objects, Unicode issues, and markdown-wrapped JSON
  • Guard: if all Jira tickets fail to fetch, generation is aborted before the AI call
📄
PDF Output Validation (PdfService):
  • Each section rendered only when its corresponding AnalysisResult field is non-empty
  • Inline markdown parser converts **bold**, *italic*, headers, bullet lists to styled PDF elements
  • JSON → Markdown fallback if a field contains raw JSON instead of prose
  • File written to ./output directory, path returned to client for download

5. Guardrails, Security & Resilience
Input safety, secret management, rate-limit handling, and AI output sanitation
🔑 Secret & Credential Management
  • No secrets in code — all API tokens, passwords, and keys are stored exclusively in appsettings.json / environment variables
  • appsettings.Development.json is listed in .gitignore — developer credentials are never committed to the repository
  • appsettings.json ships with placeholder values (YOUR_GEMINI_API_KEY, YOUR_JIRA_API_TOKEN) — no real secrets in version control
  • In production: credentials should be injected via Azure Key Vault, environment variables, or a secrets manager — not baked into the image
📁 Path Traversal Prevention
  • The GET /api/documentation/download/{fileName} endpoint sanitises the file name with Path.GetFileName(fileName) before constructing the full path — prevents ../../../etc/passwd style attacks
  • The output directory is resolved with Path.GetFullPath("./output") and the final path is validated to reside within that directory
  • Git clone directory defaults to Path.GetTempPath() with a unique GUID subdirectory — no user-controlled path components
🤖 AI Input Guardrails
  • Empty source guard: if all requested Jira tickets fail to fetch, the generation pipeline is aborted before any AI call is made — prevents sending empty context that causes hallucinated content
  • Prompt truncation: a hard PromptMaxChars ceiling caps the entire assembled prompt. Any overflow is cut and [Source data truncated] is appended so the AI knows context was limited
  • Per-field truncation: each source data field (description, comments, file content) is individually capped before assembly — prevents a single large field from consuming the entire budget
  • User Guide status gate: User Guide generation is blocked if any selected Jira ticket is not in Done/Complete status — ensures the guide reflects finished, shipped work
📤 AI Output Guardrails
  • NormalizeResult: strips placeholder text, "N/A" values, and empty collections from the AI response before PDF generation
  • RedistributeSections: rescues content misrouted by the AI (e.g. technical details placed in the executive summary)
  • 6-strategy JSON fallback pipeline: handles malformed JSON, nested structures, Unicode line separators, and markdown-wrapped responses — no single parse failure crashes the pipeline
  • Conditional PDF rendering: sections with empty content are silently skipped — the output document never contains blank or placeholder sections
🔁 Rate Limiting & Resilience
  • 429 auto-retry: up to 3 retries on Gemini quota-exceeded responses. Uses retryDelay from the Gemini JSON error body if present, otherwise waits 5s + 1s buffer
  • 5xx exponential backoff: server errors retry at 1.5s → 4s → 9s intervals before propagating
  • Per-call timeout: each Gemini API call is wrapped in a CancellationTokenSource with TimeoutSeconds (default 300s) — hangs never block indefinitely
  • Polling cleanup: the JobStatusService auto-expires stale job entries after 30 minutes and purges them on every write
  • Git clone cleanup: cloned repositories are deleted after analysis regardless of success or failure — temp storage is never leaked
🌐 CORS & Network Security
  • CORS policy is configured via Cors:AllowedOrigins in appsettings.json — no wildcard * in production
  • In development the Blazor WASM origin (http://localhost:5XXX) is the only allowed origin
  • All external API calls (Gemini, Jira, GitHub) are made server-side from the API — the browser never holds credentials or makes direct LLM calls
