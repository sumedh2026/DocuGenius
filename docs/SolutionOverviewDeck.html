🎯 Problem Statement
Why does documentation remain the biggest bottleneck in software delivery?
⏱️ Writing Docs Takes Too Long
  • Developers spend 20–30% of sprint time writing documentation manually.
  • Each JIRA ticket requires separate write-ups for technical specs, acceptance criteria, and release notes.
  • Git repository analysis — cloning, reading code, summarising flows — takes hours per project.
  • Repetitive formatting and structuring effort is wasted sprint after sprint.
🔄 Docs Go Stale Instantly
  • Code changes every day; documentation rarely keeps up.
  • Outdated docs mislead new developers and cause incorrect assumptions.
  • No automated link between JIRA tickets and technical write-ups.
  • Teams maintain multiple versions of the same document in different places.
📉 No Standardisation
  • Every developer writes documentation differently — no consistent tone, structure, or depth.
  • Technical docs, user guides, API references, and architecture overviews each need different templates.
  • New team members cannot predict where or how documentation is stored.
  • Clients and stakeholders receive inconsistent quality across deliverables.
🚪 Slow Developer Onboarding
  • New joiners spend 2–4 weeks just understanding the existing codebase.
  • Missing or outdated README files leave developers exploring code blindly.
  • No single source of truth that explains architecture, APIs, and setup steps together.
  • Senior developers waste time answering repetitive onboarding questions.

💡 Our Approach
How DocuGenious solves the documentation problem — step by step.
🔌 Step 1 — Connect to Existing Tools
  • Integrate with Atlassian JIRA — fetch ticket descriptions, acceptance criteria, comments, and linked issues automatically via JIRA REST API.
  • Integrate with GitHub / Git — clone any public or private repository, walk the file tree, and read source files without any manual setup.
  • Both sources can be used together — combine JIRA context with actual code for richer, more accurate documentation.
  • No data export or copy-paste required — works directly from live systems.
🤖 Step 2 — AI-Powered Analysis
  • Google Gemini AI (gemini-2.5-flash) reads and understands all fetched content in a single pass.
  • The AI identifies architecture patterns, API endpoints, dependencies, and business logic automatically.
  • Prompts are tuned per document type — technical docs get different instructions than user guides.
  • Output is always structured JSON mapped to a fixed schema — consistent every time.
📄 Step 3 — Generate Multiple Document Types
  • Technical Documentation — deep technical detail for developers.
  • API Documentation — endpoint reference with methods, paths, request/response bodies.
  • Architecture Overview — system design, component interactions, and data flow.
  • User Guide — plain-English usage instructions for non-technical audiences.
  • Full Documentation — all of the above in one comprehensive document.
📥 Step 4 — Deliver as Professional PDF
  • Every document is rendered as a branded, print-ready PDF using QuestPDF.
  • Consistent formatting — headings, bullet lists, code blocks, API tables, and feature cards.
  • Immediate download from the web UI or CLI — no manual formatting needed.
  • File history is preserved — previous documents are always available for re-download.

🏗️ Solution Design
A modular, cloud-ready .NET 10 architecture — each concern is isolated in its own project.
🌐
DocuGenious.Blazor
Blazor WebAssembly frontend. Source selection, ticket input, document type picker, live progress dialog, file history & PDF download.
↓
⚙️
DocuGenious.Api
.NET 10 REST API. Orchestrates all integrations, manages job status streaming, exposes PDF download endpoint. Hosted on Azure App Service.
↓
💻
DocuGenious.Console
CLI tool for power users and CI/CD pipelines. Supports all generation modes with Rich terminal progress display.
🤖
Integration.Gemini
Google Gemini AI via REST. Handles prompting, 6-strategy JSON repair, 429 auto-retry with backoff, and context-window management.
📋
Integration.Jira
Atlassian JIRA REST API v3. Fetches ticket details, subtasks, acceptance criteria, linked issues, and comments.
🗂️
Integration.Git
LibGit2Sharp clone + file walker. Works on Azure App Service using temp directory. Auto-cleanup after analysis.
📄
Integration.Pdf
QuestPDF renderer. Full markdown parser — headings, bullets, code blocks, API tables, inline bold/italic/code.
🔩
DocuGenious.Core
Shared models, interfaces, and configuration (AppSettings, AnalysisResult, IGroqService, IPdfService). No external dependencies.
🔑 Key Design Decisions
  • No AI SDK dependency — Gemini called via plain HttpClient REST.
  • Fixed JSON schema — AI output always mapped to AnalysisResult model.
  • 6-strategy JSON repair pipeline handles malformed AI responses gracefully.
  • Per-request GUID subdirectories prevent concurrency issues during Git clone.
☁️ Cloud & Deployment
  • Hosted on Azure App Service — auto-scales, no infrastructure management.
  • Git clone uses Path.GetTempPath() — works on Azure, Docker, and Linux.
  • Secrets managed via Azure App Settings (not in source code).
  • Blazor WASM served as static files — no server-side rendering needed.
