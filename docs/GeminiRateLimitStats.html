3

HTTP 5xx — Server Error Retries
3
Up to 3 retries for Gemini server errors with increasing backoff delays: 1.5s → 4s → 9s.
0
Initial request sent — DocuGenious calls Gemini API with full prompt
t = 0 s
429
Rate limit hit — Gemini returns HTTP 429. Response JSON body contains "retryDelay": "Xs" field. DocuGenious reads this delay value via ExtractRetryDelay().
Gemini recommends: 4–10 s
R1
Auto-retry #1 — Waits retryDelay + 1s buffer (default: 5s if no delay in response), then resends the same request automatically.
Wait: ~5–11 s
R2

Wait: ~5–11 s

🔽 Truncation Funnel — How DocuGenious Reduces 1 MB to ~20,000 Tokens

DocuGenious applies three progressive truncation layers before sending anything to Gemini. Each layer is sourced directly from GeminiService.cs.

L1
Per-File Content Limit
Each source file is truncated to 2,000 characters regardless of actual file size. A 50 KB file and a 5 KB file both contribute the same 2,000 chars to the prompt. This means a 1 MB file is cut to 2,000 chars — a 99.8% reduction.
Code: Truncate(f.Content!, 2000) in BuildGitContext()
Input
1,048,576 chars
↓
2,000 chars / file
L2
Maximum Files Processed
Even if the repository contains hundreds of files, DocuGenious only processes the first 20 files. Combined with Layer 1: maximum file content = 20 × 2,000 = 40,000 chars ≈ 10,000 tokens. A 1 MB repo spread across 100 files would have 80 files completely ignored.
Code: repo.Files.Where(x => x.Content != null).Take(20)
Max files
Unlimited
↓
20 files max
L3
Overall Prompt Hard Cap
The complete assembled prompt (system + user + all file contents + metadata) is capped at 80,000 characters (~20,000 tokens) as a final safety net. This covers cases where many small files together approach the limit, or combined JIRA + Git scenarios. If the prompt exceeds 80K chars, it is cut with a [Source data truncated] notice appended.
Code: TruncateIfNeeded(prompt) → prompt.Length > 80_000 ? prompt[..80_000] + "..." : prompt
